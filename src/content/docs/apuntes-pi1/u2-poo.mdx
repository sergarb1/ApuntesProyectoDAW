## Por qué diseñar antes de codificar

Diseñar la solución es como hacer los planos de una casa antes de construirla.
Puedes empezar a poner ladrillos directamente, pero probablemente te falten
ventanas, los pasillos no lleven a ninguna parte, y el tejado se caiga.

Diseñar tu proyecto significa responder:

- ¿Qué **partes** o **módulos** tiene mi aplicación?
- ¿Cómo se **comunican** entre sí?
- ¿Qué **responsabilidades** tiene cada una?
- ¿Qué **datos** necesita cada parte?

## Divide y vencerás

Todo proyecto se puede dividir en partes más pequeñas. La clave está en
elegir **cómo** dividirlo.

### Por capas (la más común)

```
Interfaz de usuario  ←  lo que ve el usuario
       ↓
Lógica de negocio   ←  las reglas de tu aplicación
       ↓
Acceso a datos      ←  cómo guardas y recuperas información
```

Cada capa se comunica solo con la de al lado. Así, si mañana cambias
de base de datos, solo tocas la capa de datos. La interfaz sigue igual.

### Por funcionalidades

```
├── Usuarios      (registro, login, perfil)
├── Productos     (listar, crear, editar, eliminar)
├── Pedidos       (crear, ver historial, estados)
└── Pagos         (simulación de pago, factura)
```

Cada funcionalidad es independiente. Puedes trabajar en una sin romper otra.

<Aclaracion pregunta="¿Y si mi proyecto es pequeño y no necesita tanta estructura?">
  Todos los proyectos empiezan siendo pequeños. Pero el que crece sin
  estructura se convierte en un caos imposible de mantener.

  No necesitas 15 capas ni patrones complejos. Con separar en 3-4 partes
  bien definidas tienes suficiente. Lo importante es que cada parte tenga
  una **responsabilidad clara** y no se mezclen.

  Piensa en ello como los cajones de un armario: aunque tengas poca ropa,
  poner calcetines con calcetines y camisetas con camisetas te ahorrará
  dolores de cabeza cuando tengas que encontrar algo.
</Aclaracion>

## Cómo documentar el diseño

No necesitas un documento de 50 páginas. Con estos diagramas basta:

### Diagrama de arquitectura (cajas y flechas)

Dibuja las partes de tu aplicación y cómo se conectan. Herramientas:
**Draw.io** (online, gratis), **Mermaid** (en Markdown), o lápiz y papel.

```
[Navegador] → [Servidor web] → [API] → [Base de datos]
                                  ↓
                          [Servicio de correo] (para emails)
```

### Diagrama de componentes

Para cada parte, enumera qué hace y qué necesita:

| Componente | Responsabilidad | Depende de |
|-----------|----------------|------------|
| Controlador de usuarios | Gestionar registro y login | Servicio de usuarios |
| Servicio de usuarios | Validar datos, encriptar passwords | Repositorio de usuarios |
| Repositorio de usuarios | Guardar/recuperar de BD | Base de datos |

## Decisiones técnicas con criterio

Cuando diseñas tu solución, tendrás que elegir tecnologías. Hazlo con criterio:

1. **No elijas lo que está de moda**, elige lo que resuelve tu problema
2. **No uses una tecnología solo porque la conoces**, a veces merece la pena aprender algo nuevo
3. **No cambies de tecnología a mitad del proyecto** a menos que sea estrictamente necesario

Para cada decisión, pregúntate:
- ¿Esta tecnología me ayuda a resolver el problema o me añade complejidad?
- ¿Tengo tiempo para aprenderla o voy a estresarme?
- ¿El equipo (si lo hay) la conoce o vamos a perder tiempo?

<CasoPractico titulo="El equipo que no diseñó nada">
  Cuatro estudiantes decidieron hacer una app para alquilar apuntes entre
  alumnos. Se repartieron el trabajo así:
  - Persona A: pantalla de login
  - Persona B: pantalla de subir apuntes
  - Persona C: pantalla de buscar
  - Persona D: base de datos

  Empezaron a codificar el día 1. A las dos semanas, Persona A había hecho
  el login en React. Persona B había hecho "subir apuntes" en HTML plano.
  Persona C había hecho la búsqueda con PHP. Persona D había creado una
  base de datos en MySQL pero sin conectarla a nada.

  Cuando intentaron unirlo todo, nada funcionaba. Cada uno había usado
  tecnologías distintas, estilos diferentes, y los datos no se comunicaban.

  **Lección**: diseñar la solución juntos, **antes** de escribir código,
  habría evitado meses de frustración. Una tarde dibujando la arquitectura
  habría ahorrado tres semanas de trabajo inútil.

  **Qué harían hoy**: dedicar 2 días a decidir stack común, definir
  la estructura de carpetas, acordar cómo se comunican las partes, y
  hacer un prototipo esqueleto antes de que cada uno trabaje por su lado.
</CasoPractico>

## Checklist para tu proyecto

Antes de escribir la primera línea de código:

- [ ] He dividido el proyecto en 3-7 partes o módulos
- [ ] Cada parte tiene una responsabilidad clara
- [ ] He dibujado un diagrama de arquitectura
- [ ] He elegido las tecnologías con criterio
- [ ] El equipo (si existe) está de acuerdo con la estructura
- [ ] He validado el diseño con el profesor
