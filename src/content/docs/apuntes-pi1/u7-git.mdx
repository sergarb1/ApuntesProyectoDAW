## El equipo empieza antes que el código

Antes de escribir una línea, el equipo debe acordar:

### Reglas básicas

1. **Herramientas de comunicación**: un grupo de WhatsApp o Discord es mínimo. Pero
   las decisiones importantes no se toman por chat, se hablan.
2. **Reparto de tareas**: quién hace qué. Que sea claro y equilibrado.
3. **Reuniones periódicas**: aunque sean 10 minutos al día para actualizar el estado.
4. **Toma de decisiones**: ¿cómo se decide algo si hay desacuerdo?

### Flujo de trabajo con git

El flujo básico para un equipo pequeño:

```
main          → solo versiones estables y funcionales
  └─ dev      → rama de desarrollo, donde se integra todo
       ├─ feature/login   → cada funcionalidad en su rama
       ├─ feature/productos
       └─ feature/pedidos
```

**Reglas**:
- Nunca commits directamente a `main`
- Cada funcionalidad nueva → rama nueva desde `dev`
- Antes de mergear: code review por otro miembro del equipo
- Merge a `dev` → probar → merge a `main`

<Aclaracion pregunta="¿Y si somos dos y no necesitamos tanto flujo?">
  Aunque seáis dos, el flujo de ramas os protege. Trabajar directamente
  sobre `main` es la receta para el desastre: "es que iba a arreglar una
  cosa y se me borró todo".

  Para equipos de 2-3 personas, basta con:
  - `main` (lo que funciona)
  - Una rama por persona o por funcionalidad
  - Commits pequeños con mensajes descriptivos
  - Comunicación diaria de "esto estoy haciendo, esto voy a hacer"
</Aclaracion>

## Código limpio en equipo

### Estilo común
Acordad un estilo de código al principio:
- ¿Tabulaciones o espacios?
- ¿Punto y coma o no?
- ¿Comillas simples o dobles?
- ¿Nombres en español o inglés?

Usad herramientas como **ESLint** o **Prettier** para que el código
de todos tenga el mismo formato automáticamente.

### Code review

Revisad el código de los demás antes de mergear. No es para "cazar
errores", es para:
- Aprender cómo trabaja tu compañero
- Detectar malos entendidos ("ah, yo pensaba que esta función hacía otra cosa")
- Mejorar la calidad entre todos

### Commits que se entienden

```bash
# ❌ Mal
git commit -m "cambios"

# ✅ Bien
git commit -m "Añadir validación de email en formulario de registro"
```

Un buen mensaje de commit explica **qué cambia y por qué**.

  Tres estudiantes empezaron el proyecto con entusiasmo. Cada uno se llevó
  una parte a casa. Dos semanas después:
  - Persona A había hecho el login y el registro
  - Persona B había hecho la gestión de productos
  - Persona C se había desconectado y no respondía

  En la reunión con el profesor, Persona A y B explicaron la situación.
  El profesor les sugirió: "repartid las tareas de Persona C entre los dos,
  pero también haced que él se responsabilice de algo concreto y medible".

  Se sentaron, hablaron claro, y rediseñaron el reparto:
  - Persona A: frontend completo (que ya había mostrado interés)
  - Persona B: backend completo
  - Persona C: documentación + tests + despliegue (responsabilidades claras)

  A partir de ahí, se reunían 15 minutos cada día (por Discord) para
  actualizar el progreso. No funcionó perfecto, pero funcionó.

  **Lección**: un equipo no es "cada uno a su bola". Es comunicación
  constante, reparto claro de responsabilidades, y saber reaccionar
  cuando alguien se queda atrás.
</CasoPractico>

## Conflictos (de código y de personas)

### Conflictos en git

Los conflictos ocurren cuando dos personas modifican la misma parte.
No pasa nada. Así se resuelven:

1. `git pull` la rama destino
2. Busca los marcadores `<<<<<<<`, `=======`, `>>>>>>>`
3. Decide qué versión conservar (o una combinación)
4. Borra los marcadores
5. Commit

Con práctica, resolver conflictos lleva segundos.

### Conflictos entre personas

Si alguien no trabaja, si hay diferencias de criterio, si la comunicación
falla:

1. **Habladlo en persona** (no por chat). Cada uno explica su punto de vista.
2. **Buscad un acuerdo** o una solución intermedia.
3. Si no hay acuerdo, **tercera persona** (el profesor, un mediador).

## Checklist para tu equipo

- [ ] Hemos acordado herramientas de comunicación
- [ ] Tenemos un repositorio compartido con ramas
- [ ] Cada uno sabe qué tiene que hacer esta semana
- [ ] Nos reunimos al menos 2 veces por semana
- [ ] Hacemos code review antes de mergear
- [ ] Los commits son descriptivos y pequeños
- [ ] Tenemos un plan si alguien se queda atrás
