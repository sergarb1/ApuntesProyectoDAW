## Objetivos de la unidad

- Comprender los conceptos fundamentales de Git

- Resolver conflictos básicos

## Introducción

Git es un sistema de control de versiones distribuido que permite rastrear cambios en el código, colaborar con otros desarrolladores y mantener un historial completo del proyecto.

## Configuración inicial

```bash
# Configurar identidad (una vez)
git config --global user.name "Tu Nombre"
git config --global user.email "tu@email.com"

# Ver configuración
git config --list

# Alias útiles
git config --global alias.lg "log --oneline --graph --all"
git config --global alias.s "status -s"
```

## Flujo básico

```bash
# Inicializar repositorio
git init

# Ver estado
git status

# Añadir archivos al área de staging
git add index.html
git add .              # todos los archivos

# Confirmar cambios
git commit -m "Añadir página principal"

# Ver historial
git log                # historial completo
git log --oneline      # resumen en una línea
git log --oneline --graph  # historial gráfico
```

## El área de staging

```
Working Directory → git add → Staging Area → git commit → Repositorio
```

```bash
# Añadir interactivamente
git add -p

# Ver cambios no añadidos
git diff

# Ver cambios en staging
git diff --staged

# Quitar archivo de staging (sin perder cambios)
git reset HEAD archivo.txt

# Deshacer cambios locales (restaurar versión del último commit)
git checkout -- archivo.txt
```

## Ramas (branches)

Las ramas permiten desarrollar funcionalidades de forma aislada.

```bash
# Crear rama
git branch feature/login

# Cambiar a otra rama
git checkout feature/login
# o crear y cambiar en un paso:
git checkout -b feature/login

# Listar ramas
git branch                 # locales
git branch -a              # todas (incluye remotas)

# Fusionar rama (estando en main/master)
git checkout main
git merge feature/login

# Eliminar rama
git branch -d feature/login     # local
git branch -D feature/login     # forzar si no está fusionada
```

### Merge vs Rebase

```bash
# Merge: crea un commit de fusión (preserva historial)
git checkout main
git merge feature

# Rebase: reescribe historial (historial lineal)
git checkout feature
git rebase main
```

## Remotos (GitHub)

```bash
# Conectar repositorio local con remoto
git remote add origin https://github.com/usuario/repo.git

# Ver remotos
git remote -v

# Subir cambios
git push -u origin main    # primera vez
git push                   # siguientes

# Bajar cambios
git pull                   # fetch + merge
git fetch                  # solo descarga, no fusiona

# Clonar repositorio existente
git clone https://github.com/usuario/repo.git
```

## Pull Requests (GitHub)

Flujo de trabajo típico con PRs:

```bash
# 1. Actualizar main
git checkout main
git pull origin main

# 2. Crear rama para la funcionalidad
git checkout -b feature/nueva-funcion

# 3. Hacer cambios y commits
git add .
git commit -m "Añadir nueva función"

# 4. Subir rama a GitHub
git push origin feature/nueva-funcion

# 5. En GitHub: crear Pull Request desde feature/nueva-funcion → main

# 6. Después de que el PR se apruebe, eliminar rama local
git checkout main
git pull origin main
git branch -d feature/nueva-funcion
```

## Resolver conflictos

Los conflictos ocurren cuando dos ramas modifican las mismas líneas.

```bash
# Al hacer merge, Git muestra:
# Auto-merging index.html
# CONFLICT (content): Merge conflict in index.html

# El archivo contendrá:
# <<<<<<< HEAD
# <h1>Título desde main</h1>
# =======
# <h1>Título desde feature</h1>
# >>>>>>> feature
```

Para resolver:
1. Editar el archivo, quedándote con el contenido deseado
2. Eliminar los marcadores `<<<<<<<`, `=======`, `>>>>>>>`
3. Guardar, añadir y commitear

```bash
git add index.html
git commit -m "Resolver conflicto en index.html"
```

## .gitignore

Archivo que excluye archivos/directorios del control de versiones:

```txt
# Dependencias
node_modules/
vendor/

# Build
dist/
build/
.env

# IDE
.vscode/
.idea/

# Sistema
.DS_Store
Thumbs.db

# Logs
*.log
```

## Buenas prácticas con Git

1. **Commits atómicos**: cada commit hace una sola cosa
2. **Mensajes descriptivos**: `"Añadir validación de email en formulario de registro"` no `"cambios"`
3. **Commits frecuentes**: mejor muchos pequeños que uno gigante
4. **No commitear archivos generados**: usa `.gitignore` para builds, dependencias, .env
5. **Ramas por funcionalidad**: `feature/`, `bugfix/`, `hotfix/`
6. **Pull antes de push**: `git pull --rebase` para mantener historial lineal

## Ejercicio práctico

Simula el flujo colaborativo:
1. Crea un repositorio local con `git init`
2. Haz 3 commits con archivos distintos (index.html, style.css, app.js)
3. Crea una rama `feature/nueva-seccion` y añade un archivo
4. Vuelve a main y haz otro commit
5. Fusiona la rama feature con main

<details>
<summary>Solución paso a paso</summary>

```bash
# 1. Inicializar
mkdir proyecto-ejemplo
cd proyecto-ejemplo
git init

# 2. Primer commit
echo "<h1>Inicio</h1>" > index.html
git add index.html
git commit -m "Añadir index.html"

# 3. Segundo commit
echo "body { font-family: sans-serif; }" > style.css
git add style.css
git commit -m "Añadir styles básicos"

# 4. Tercer commit
echo "console.log('App iniciada');" > app.js
git add app.js
git commit -m "Añadir script inicial"

# 5. Rama feature
git checkout -b feature/nueva-seccion
echo "<section><h2>Nueva sección</h2></section>" >> index.html
git add index.html
git commit -m "Añadir nueva sección al HTML"

# 6. Volver a main y hacer commit
git checkout main
echo "/* Comentario */" >> style.css
git add style.css
git commit -m "Añadir comentario a CSS"

# 7. Fusionar
git merge feature/nueva-seccion
git log --oneline --graph
```

</details>

## Resumen

- `git init` crea un repositorio
- `git add` + `git commit` guardan versiones del código
- Las **ramas** permiten desarrollo paralelo
- `git merge` o `git rebase` integran cambios
- Los **pull requests** facilitan la revisión de código
- Los conflictos se resuelven manualmente editando archivos
