## Por qué planificar (aunque te dé pereza)

Planificar no es hacer documentos bonitos. Es **responder estas preguntas**
en cada momento:

- ¿Qué tengo que hacer esta semana?
- ¿Cuánto tiempo me llevará?
- ¿Qué hago si me retraso?
- ¿Estoy avanzando al ritmo correcto?

Sin planificación, avanzas a ciegas. Con planificación, sabes si vas bien,
si necesitas acelerar, o si tienes que recortar funcionalidades.

## Métodos para organizarte

### Kanban (el más visual)

Un tablero con columnas: **Por hacer → En proceso → Hecho**.

Cada tarea es una tarjeta que se mueve de izquierda a derecha.

| Por hacer | En proceso | Hecho |
|-----------|------------|-------|
| CRUD usuarios | Login (Javi) | Diseño BD |
| Pantalla inicio | Perfil (Marta) | Prototipo |
| Tests | — | — |

**Herramientas**: Trello, GitHub Projects, Notion, un pizarra física.

### SCRUM adaptado a 2-3 personas

No necesitas un Scrum Master ni sprints de 2 semanas perfectos. Pero sí:

1. **Sprint**: bloque de 1-2 semanas con objetivos concretos
2. **Reunión de planificación**: qué vamos a hacer este sprint
3. **Reunión diaria** (15 min): qué hice ayer, qué haré hoy, bloqueos
4. **Revisión**: al final del sprint, qué se consiguió

<Aclaracion pregunta="¿No es demasiado para un proyecto de clase?">
  Para un proyecto de clase pequeño, igual sí. Pero para un proyecto
  intermodular que dura meses y en equipo, la planificación ligera
  es **necesaria**.

  No hace falta ceremonies formales. Basta con:
  - Un tablero Kanban (Trello, GitHub Projects)
  - Reuniones de 10-15 minutos, 2 veces por semana
  - Objetivos claros para cada semana

  La planificación no es burocracia. Es un **mapa** para no perderte.
</Aclaracion>

## Cómo estimar bien

Estimar es difícil porque siempre tendemos al optimismo.
"Esto lo hago en 2 días" → en realidad son 5.

### Técnica: desglose y multiplica

1. Divide la tarea en subtareas de máximo 4 horas
2. Estima cada subtarea
3. Multiplica el total por **2** (para cubrir imprevistos)

**Ejemplo**: "Hacer pantalla de registro"

| Subtarea | Estimación |
|----------|-----------|
| Maquetar HTML | 2h |
| Estilos CSS | 1h |
| Validación JS | 2h |
| Conectar con API | 2h |
| Probar | 1h |
| **Total** | **8h → 16h (2 días)** |

El multiplicador por 2 no es pesimismo. Es experiencia. Los imprevistos,
las dudas, los "y si...", las interrupciones ocurren siempre.

## Cómo recuperar el tiempo perdido

Si te das cuenta de que no llegas (y te darás cuenta si planificas):

1. **Prioriza**: haz solo lo que realmente importa para el MVP
2. **Recorta**: lo que no sea imprescindible, fuera
3. **Simplifica**: una funcionalidad simple que funciona > una compleja a medias
4. **Pide ayuda**: al profesor o a compañeros
5. **No te engañes**: si ves que no llegas, actúa, no esperes

<CasoPractico titulo="El proyecto que se salvó por un tablero Kanban">

  Un equipo de tres empezó usando solo un grupo de WhatsApp para organizarse.
  "Hoy hago esto", "mañana haré lo otro". Sin prioridades claras, sin fechas.

  A mitad de curso, un profesor les recomendó usar **GitHub Projects**.
  Se tomaron una tarde para:
  - Listar todas las tareas pendientes
  - Priorizarlas (imprescindible / deseable / futuro)
  - Asignar responsables
  - Poner fechas límite

  Lo que descubrieron les sorprendió: tenían el doble de tareas de las
  que pensaban y la mitad del tiempo para hacerlas. Recortaron funcionalidades,
  reordenaron prioridades, y se centraron en lo importante.

  **Resultado**: entregaron a tiempo, con un proyecto completo y funcional.
  No sobraron lujos, pero no faltó lo esencial.

  **Lección**: ver el trabajo reflejado en un tablero no estresa. Libera.
  Porque lo que no está escrito no existe, y lo que no está planificado
  no se hace.
</CasoPractico>

## Checklist para tu planificación

- [ ] Tengo un tablero con todas las tareas del proyecto
- [ ] Cada tarea tiene responsable y fecha estimada
- [ ] Tengo las funcionalidades priorizadas (MVP vs deseable)
- [ ] Sé lo que tengo que hacer esta semana
- [ ] Hago reuniones de equipo al menos cada 3-4 días
- [ ] Si veo que me retraso, ajusto el plan inmediatamente
- [ ] Cada sprint entrego algo funcional, no "código a medias"
