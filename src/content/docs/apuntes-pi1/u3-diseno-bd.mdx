## Piensa en los datos antes de programar

Antes de escribir una sola función, pregúntate:

- ¿Qué **entidades** existen en mi proyecto? (usuarios, productos, pedidos...)
- ¿Qué **atributos** tiene cada una? (nombre, email, precio, fecha...)
- ¿Cómo se **relacionan** entre sí? (un usuario tiene muchos pedidos)
- ¿Qué **reglas** debo cumplir? (email único, precio positivo...)

### De las ideas a las entidades

Coge tu idea de proyecto y extrae los **sustantivos**. Esos son tus candidatos
a entidades:

> "Una aplicación donde **usuarios** pueden publicar **recetas**,
> y otros **usuarios** pueden dejar **comentarios** y **valoraciones**"

Entidades: `Usuario`, `Receta`, `Comentario`, `Valoracion`

### Relaciones entre entidades

Conéctalas. Pregunta: "¿cuántos?":

- Un `Usuario` tiene muchas `Recetas` → 1 a N
- Una `Receta` tiene muchos `Comentarios` → 1 a N
- Un `Usuario` deja muchas `Valoraciones` → 1 a N
- Una `Receta` tiene muchas `Valoraciones` → 1 a N

<Aclaracion pregunta="¿Necesito siempre una base de datos relacional?">
  No. Depende de lo que guardes:

  - **Datos estructurados con relaciones** (usuarios ↔ pedidos) → SQL
  - **Documentos flexibles** (catálogos, configuraciones) → NoSQL
  - **Archivos** (imágenes, PDFs) → Sistema de archivos o cloud storage
  - **Datos temporales** (sesiones, caché) → En memoria (Redis, etc.)

  Para un proyecto DAW, lo normal es SQL (MySQL, PostgreSQL, SQLite).
  Es lo que se enseña, lo más documentado, y suficiente para el 90% de los casos.
</Aclaracion>

## Del modelo conceptual al esquema

No necesitas ser experto en bases de datos para hacer un buen diseño.
Sigue estos pasos:

### Paso 1: Dibuja las entidades y relaciones

Con lápiz y papel o Draw.io:
```
[Usuario] ───< tiene >─── [Receta]
   │                          │
   └──< deja >── [Comentario] ┘
```

### Paso 2: Define los atributos clave

Para cada entidad, decide qué datos guardar. Empieza por los imprescindibles
y añade solo lo que realmente necesitas:

| Entidad | Atributos |
|---------|-----------|
| Usuario | id, nombre, email, password_hash, fecha_registro |
| Receta | id, titulo, ingredientes, instrucciones, usuario_id, fecha |
| Comentario | id, texto, usuario_id, receta_id, fecha |

### Paso 3: Traduce a tablas

Cada entidad → una tabla. Cada atributo → una columna.
Las relaciones 1 a N → una clave foránea en la tabla "muchos".

## Buenas prácticas para tus datos

### Nombra bien las cosas
- Tablas en plural y snake_case: `usuarios`, `recetas_favoritas`
- Columnas descriptivas: `fecha_creacion`, `precio_unitario`
- Clave primaria siempre `id`

### No guardes lo que puedas calcular
- Guarda `fecha_nacimiento`, no `edad` (la edad se calcula)
- Guarda `precio_unitario` y `cantidad`, no `precio_total` (se calcula)

### Protege los datos sensibles
- Contraseñas → hash (bcrypt), nunca texto plano
- Emails → pueden ser identificativos, trátalos con cuidado
- Datos bancarios → no los guardes si no es estrictamente necesario

<CasoPractico titulo="Una base de datos que creció sin control">
  Un equipo hizo una aplicación para gestionar eventos. Empezaron con una
  tabla `eventos` con columnas básicas. Según avanzaba el proyecto, fueron
  añadiendo:

  - `categoria` (texto libre → cada uno escribía "música", "Musica", "MUSICA")
  - `organizador` (texto → a veces nombre, a veces email, a veces ID)
  - `asistentes` (texto → lista separada por comas: "1,2,3,4")
  - `imagenes` (otra tabla que no se llegó a conectar bien)

  El resultado: datos inconsistentes, consultas lentas, y bugs difíciles
  de rastrear. Tuvieron que rehacer la base de datos a mitad de proyecto.

  **Lección**: media hora diseñando el modelo de datos al principio habría
  ahorrado semanas de trabajo. No hace falta ser experto, solo ordenado.
</CasoPractico>

## Preguntas clave para tu proyecto

- ¿Qué pasaría si esta tabla tuviera 10.000 filas?
- ¿Un usuario puede tener 0 elementos de esta entidad?
- ¿Qué reglas debo cumplir (email único, precio mínimo)?
- ¿Qué pasa si elimino un usuario: cascada o solo ese registro?
- ¿Necesito que algún campo sea único o combinación única?
