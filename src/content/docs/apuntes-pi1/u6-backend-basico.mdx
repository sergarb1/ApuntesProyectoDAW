## El arte de empezar a programar

El mayor obstáculo no es la complejidad técnica, es saber **por dónde
empezar**. La clave está en priorizar.

### Prioriza por valor, no por dificultad

Haz primero lo que **más valor aporta al proyecto** y que **menos dependencias
tiene**. Normalmente, eso significa:

1. **Autenticación básica** — sin login, no hay usuario que use la app
2. **CRUD de la entidad principal** — la operación más básica sobre tu recurso clave
3. **Interfaz mínima** — lo justo para que se vea algo en pantalla
4. **Mejoras iterativas** — refinar lo que ya funciona

### Empieza con un esqueleto

No implementes todas las funcionalidades a la vez. Crea una ruta que
devuelva un "Hola mundo". Luego añade autenticación. Luego añade la
primera entidad. Ve creciendo por capas.

```
Iteración 1: "Hola mundo" desde el servidor
Iteración 2: Endpoint GET /api/usuarios (datos mock)
Iteración 3: Conexión a base de datos real
Iteración 4: CRUD completo de usuarios
Iteración 5: Autenticación JWT
Iteración 6: CRUD de la siguiente entidad
... así hasta terminar
```

<Aclaracion pregunta="¿Qué hago si me atasco en una funcionalidad?">
  Primero, **no te quedes bloqueado**. Si llevas más de 30 minutos sin
  avanzar, prueba esto:

  1. Divide el problema en partes más pequeñas
  2. Busca en Google o Stack Overflow (todos lo hacen)
  3. Pregunta a un compañero o al profesor
  4. **Salta a otra cosa** y vuelve después

  A veces la solución aparece cuando dejas de pensar en el problema.
  Si aún así no avanzas, simplifica la funcionalidad o pregúntate
  si realmente la necesitas en tu MVP.
</Aclaracion>

## Organiza tu código

### Estructura de carpetas

No hay una estructura única, pero esta es una base sólida:

```
src/
├── routes/          # Definición de endpoints
├── controllers/     # Lógica de cada endpoint
├── models/          # Definición de datos
├── middleware/       # Funciones intermedias (auth, validación)
├── services/        # Lógica de negocio compleja
├── utils/           # Funciones auxiliares
└── config/          # Configuración (BD, variables de entorno)
```

### Comentarios que ayudan

Comenta el **por qué**, no el **qué**:

```javascript
// ❌ Mal: ya sé que esto suma
let total = a + b;

// ✅ Bien: explica por qué haces esta operación
// Sumamos el IVA (21%) al precio base porque la ley lo exige
let precioFinal = precioBase * 1.21;
```

## Refactoriza desde el principio

No esperes a tener todo el código escrito para mejorarlo. Cada vez que
veas que una función se repite, extráela. Cada vez que una clase crezca
demasiado, divídela.

La refactorización constante evita que el código se convierta en
"espagueti". Y es más fácil refactorizar 100 líneas que 1000.

<CasoPractico titulo="El estudiante que programó la funcionalidad más difícil primero">

  Miguel decidió que su proyecto tendría un sistema de recomendación
  usando machine learning. Pasó tres semanas implementando un algoritmo
  de recomendación que nunca llegó a funcionar bien. Para entonces,
  apenas tenía el login y el registro hechos.

  En las últimas dos semanas, tuvo que hacer a toda prisa: CRUD de
  productos, carrito de compra, pasarela de pago simulada, y la interfaz.
  Todo mal hecho, sin tests, con bugs evidentes.

  **Resultado**: suspendió. Su proyecto prometía mucho, pero entregó poco.

  **Lección**: haz primero lo que sabes que funciona, lo que aporta valor
  seguro, lo que el tribunal va a ver y probar. Las funcionalidades
  "avanzadas" son el postre, no el plato principal. Si no tienes un plato
  principal sólido, el postre no importa.

  **Qué haría hoy**: empezar con el CRUD de productos (su entidad principal),
  luego el carrito, luego la autenticación, luego la interfaz. Y solo si
  sobra tiempo, experimentar con el algoritmo de recomendación.
</CasoPractico>

## Checklist para tu implementación

- [ ] Tengo una lista priorizada de funcionalidades
- [ ] Empiezo por lo que menos dependencias tiene
- [ ] Cada funcionalidad la termino antes de pasar a la siguiente
- [ ] Refactorizo cuando veo código duplicado
- [ ] Hago commits pequeños y frecuentes (no "cambios varios")
- [ ] Pruebo cada funcionalidad antes de darla por terminada
- [ ] Si me atasco, salto a otra cosa o pido ayuda
